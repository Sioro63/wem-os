# Architecture

## The one-paragraph version

WEM OS is a **feature-modular Next.js monolith**. Routes in `app/` are thin;
each business domain lives in `features/<name>` with its own schemas,
queries, mutations and components. All reads happen in server components,
all writes go through server actions that check permissions and return a
uniform `ActionResult`. PostgreSQL holds the truth; stock and documents move
together inside transactions.

## Key decisions & tradeoffs

### Roles as an enum, not a table

`User.role` is a Prisma enum wired through Better Auth's `additionalFields`.

- ✅ Zero joins, trivially type-safe, roles checked in one pure function
- ❌ Adding a role requires a migration; no per-user permission overrides

For a 4-role internal tool this is the right trade. If custom roles are ever
needed, introduce a `Role` table and swap the implementation inside
`lib/permissions.ts` — its API (`hasPermission(role, permission)`) is the
only contract the rest of the app knows.

### Flat permission strings

Permissions are strings like `quote.decide` mapped per role. UI checks are
cosmetic; **server actions and pages re-check every time**. This means a
user with a stale UI can never perform an unauthorized mutation.

### Derived available stock

`available = physical − reserved` is computed, never stored. Reservations
increment on order creation (or quote acceptance) and both columns decrement
on completion — always in the same transaction as the status change. There
is no code path where a document status and its stock effect can diverge.

### One money function

VAT math (`12%` after discount, centavo rounding) exists once in
`features/quotations/utils/totals.ts`, imported by the quote form, order
form, both create actions, and the seed. Form previews are guaranteed to
match stored totals.

### Sequential document numbers

`Q-YYYY-####` / `SO-YYYY-####` generated by counting rows for the current
year. Under heavy concurrency two requests could race; for a small internal
team this is acceptable. The upgrade path (a `DocumentCounter` table with
`UPDATE … RETURNING`) slots into `services/document-numbers.ts` without
touching callers.

### Auth in two layers

`middleware.ts` does an **optimistic** cookie-existence check (fast, no DB
hit) for redirects. Real session validation happens in `lib/session.ts` on
every protected page and action. The middleware is UX; the session check is
security.

## Request flow (mutation)

```
Client component
  → server action ("use server")
      → requirePermission()        lib/session.ts
      → zod .parse()               features/x/schemas
      → prisma transaction         (+ stock movements where relevant)
      → logActivity()              services/activity-log.ts (never throws)
      → revalidatePath()
  ← ActionResult<T>
Client: toast success / toast error
```

## V2 extension seams

The schema file ends with a commented map of planned models. Where each
feature plugs in:

| Extension | Seam |
| --------- | ---- |
| **Purchase orders** | New `PurchaseOrder` model referencing `Supplier`; receiving increments `Inventory.physicalStock` — mirror of order completion |
| **Shipments** | `Shipment` referencing `SalesOrder`; add a `SHIPPED` step to the order status flow in `features/sales-orders/utils/status.ts` (`ORDER_NEXT_STATUS` is the single source of the state machine) |
| **Barcode scanning** | Products already have unique SKUs; add a `barcode` column and a scan input that resolves to product IDs in the quote/order forms |
| **Technician jobs** | `Job` model linking `SalesOrder` + assigned `User`; new permission group `job.*` in `lib/permissions.ts` |
| **Multi-warehouse** | `Inventory` is already per-warehouse; add warehouse pickers to the stock-movement code paths (accept/create/complete currently target the default warehouse) and a `StockTransfer` model |
| **Supplier portal** | Add a `SUPPLIER` role with a scoped permission set; suppliers see their own POs |
| **Accounting export** | All money is `Decimal(12,2)` with an append-only `ActivityLog`; add CSV/API exporters in `services/` |
| **Payments** | `Payment` model referencing `SalesOrder`; webhooks live under `app/api/` |
| **Public API** | Reuse feature query/action functions from `app/api/v1/*` route handlers — they're plain async functions |
| **Mobile app** | Better Auth supports bearer tokens; the API above becomes the backend |
